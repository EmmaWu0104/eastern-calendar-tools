# 奇門遁甲傳統置閏法階段交接摘要

## 1. 本階段定位

本階段只實作奇門遁甲傳統置閏法第一版，不實作拆補法、茅山法、無閏法，也不建立 method switch。現階段目標是讓 2027 芒種、大雪、閏大雪、冬至案例，從手動 fixture 逐步推進為「年度自動分析 -> seed 建議 -> timeline」產物。

目前 resolver 仍只覆蓋 2027 驗證範圍，尚未擴大為任意年份完整 timeline。UI、1080 盤面查表與缺盤提示也尚未接入。

## 2. 核心規則決策

- 奇門有效日與干支日切換時間使用 23:00。
- 時區使用 Asia/Taipei。
- 第一版不使用真太陽時。
- 實際節氣仍取自既有天文節氣 JSON。
- 奇門節氣與陰陽遁切換依置閏法 timeline，不直接依實際節氣瞬間切換。
- 陰陽遁切換：奇門夏至上元開始陰遁，奇門冬至上元開始陽遁。
- 符頭定義：日干為甲或己。
- 上元符頭：甲子、己卯、甲午、己酉。
- 中元符頭：甲寅、己巳、甲申、己亥。
- 下元符頭：甲辰、己未、甲戌、己丑。
- 每元 5 日，三元 15 日。
- 置閏只在芒種或大雪後判斷。
- 超神達九日才置閏。
- 閏芒種仍為陽遁，局數 6 / 3 / 9。
- 閏大雪仍為陰遁，局數 4 / 7 / 1。

## 3. 目前資料架構

- `data/qimen/qimen_yuan_ju_table.json`
  - 24 節氣三元局數表。
  - 不含 method。
  - 不含斷語。
- `data/qimen/plates`
  - 陽遁 1～9 局。
  - 陰遁 1～9 局。
  - 每局 60 個時柱 key。
  - 目前 plate value 仍為 null skeleton。

目前尚未建立 1080 盤面內容。盤面資料不由 AI 演算生成，後續要依人工整理資料填入。

## 5. 2027 驗證案例摘要

### 芒種段

- 實際芒種：2027-06-06 05:26 左右，以資料檔為準。
- 奇門芒種上元：`2027-05-29T23:00:00+08:00`。
- 2027 芒種 `chaoShenDays = 8`。
- 未達九日，不置閏。
- `2027-06-13T23:00:00+08:00` 後接奇門夏至上元。
- 2027-06-14 白天已進奇門夏至，陰遁九局，狀態為超神。

### 大雪 / 閏大雪 / 冬至段

- 實際大雪：2027-12-07 16:38 左右，以資料檔為準。
- 奇門大雪上元：`2027-11-25T23:00:00+08:00`。
- 2027 大雪 `chaoShenDays = 12`。
- 達九日，應置閏大雪。
- 閏大雪上元：`2027-12-10T23:00:00+08:00`。
- 實際冬至：2027-12-22 10:42 左右，以資料檔為準。
- 實際冬至當天仍在閏大雪下元，狀態為置閏後接氣。
- 奇門冬至上元：`2027-12-25T23:00:00+08:00`。
- 2027-12-26 白天為奇門冬至上元，陽遁一局。

## 10. 交接注意事項

- 不要直接把 resolver 擴大到所有年份，需先建立完整年度 timeline 產生與驗證。
- 不要讓 UI 直接依實際節氣切換陰陽遁。
- 不要把拆補法或其他派別混入目前傳統置閏法。
- 不要用 AI 自動生成 1080 盤內容。
- 不要修改既有 `qimen_yuan_ju_table.json`，除非發現資料錯誤且有來源。
- 不要把未追蹤的 `2Drive.ffs_gui` / `Note.txt` 納入 commit。

## 11. 本階段結論

奇門傳統置閏法第一階段已從手動 2027 fixture 推進到年度自動分析產生 2027 timeline。resolver 仍維持保守範圍，只服務 2027 驗證案例。下一階段適合先擴展完整年度 seed / timeline，再進 UI 與盤面查表。
