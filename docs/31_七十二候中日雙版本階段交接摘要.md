## 二、核心設計決策

- 七十二候時間判斷公式不變。
- 仍使用目前節氣到下一節氣的區間三等分。
- 初候 / 次候 / 末候仍採左閉右開。
- 七十二候不受 23:00 換日影響。
- 不使用真太陽時。
- 中國版與日本版只是同一個 `globalHouIndex` 上的不同候名。
- `name` / `shortName` 保留作為中國版相容欄位。
- `variants.zh` 代表中國版。
- `variants.jp` 代表日本版。
- UI 第一版固定同時顯示中 / 日，不做切換。

## 四、日本版資料來源摘要

第 27 包採用的資料來源：

- 主要來源：国立天文台 暦Wiki / 七十二候
- 交叉確認：暦生活 七十二候一覧

日本版候名以国立天文台暦Wiki「宝暦暦以降」欄為主。第一版只取漢字候名，不取讀音、不取解釋長文、不取日期範圍。

節氣 key 沿用專案既有繁體中文 key，例如 `驚蟄`、`處暑`，不改成日本來源常見的 `啓蟄`、`処暑`。字形差異已於第 27 包 notes 記錄。

本階段特別確認的字形取法：

- `globalHouIndex` 1：`東風解凍`
- `globalHouIndex` 6：`草木萌動`
- `globalHouIndex` 26：`腐草為蛍`
- `globalHouIndex` 58：`虹蔵不見`
- `globalHouIndex` 72：`鶏始乳`

## 五、seventyTwoHou.js 摘要

`src/seventyTwoHou.js` 現況：

- `getHouDefinitions()` 回傳 clone 後的七十二候定義。
- `getHouBySolarTerm(termName)` 依節氣名稱回傳 clone 後的候資料。
- `getCurrentHouBySolarTermRange(...)` 回傳目前候。
- `getNextHouBySolarTermRange(...)` 回傳下一候。
- `currentHou` / `nextHou` 現在都會帶出 `variants.zh` / `variants.jp`。
- 回傳資料仍保留既有 `name` / `shortName` / `description`。
- `createHouResult(...)` 會 clone 候資料後再組裝結果。
- `normalizeHouVariants(...)` 提供相容保護：
  - `zh` 缺漏時由 `name` / `shortName` 補出。
  - `jp` 缺漏時回 `null`。
  - 不讓查詢流程 throw。

本階段沒有改七十二候時間公式，也沒有改 bazi 整合邏輯。bazi 透過既有七十二候查詢自然取得 `variants`。

## 六、UI 摘要

`src/main.js` 與 `styles/main.css` 目前將七十二候顯示為中 / 日雙版本。

顯示格式：

```text
七十二候
中：立春初候・東風解凍
日：立春初候・東風解凍

候區間：02/04 04:02 ～ 02/09 02:38

下一候
中：立春次候・蟄蟲始振
日：立春次候・黄鶯睍睆
```

- `currentHou` 顯示七十二候中 / 日雙版本。
- `nextHou` 顯示下一候中 / 日雙版本。
- 候區間只顯示一組，因為中日版共用同一時間區間。
- `zh` 缺漏時 fallback 既有中國版 `shortName` / `name`。
- `jp` 缺漏時顯示 `—`。
- `currentHou` / `nextHou` 為空時不 throw。
- 樣式採垂直排列，手機版避免水平 overflow。
- 不影響九宮飛星、金函玉鏡、每日資訊、建除十二神等既有區塊。
