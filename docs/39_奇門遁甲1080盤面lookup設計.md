# 39 奇門遁甲 1080 盤面 lookup 設計

本文件是第 68 包 docs-only 設計文件，目標是設計 1080 盤面 lookup helper 的資料契約與 fallback 行為。

本包不實作 helper、不修改 plate JSON、不接 UI、不接手動覆寫、不改 resolver / formatter。目前 1080 plate JSON 仍是 skeleton / null plates；定局成功不代表盤面資料已完成。盤面 lookup 是 formatter 後的一層，不應回寫 formatter 結果。

## 1. 目前 1080 plate 資料結構

目前資料路徑：

```text
data/qimen/plates/{dunType}/ju-{ju}.json
```

例如：

```text
data/qimen/plates/yang/ju-1.json
data/qimen/plates/yin/ju-9.json
```

每個檔案目前預期包含：

* `meta`
* `plates`

其中 `meta` 包含：

* `schemaVersion`
* `dunType`
* `dunName`
* `ju`
* `plateCount`

`plates` 目前應有 60 個干支時柱 key，例如：

* `甲子`
* `乙丑`
* ...
* 60 甲子時柱

目前每個 plate value 仍是 `null` skeleton。

`validateQimenData(...)` 已檢查 18 個 plate files，且每個檔案都有 60 個 plates key。目前 null plates 數量代表盤面尚未填入。null plate 是資料未完成，不是 formatter 失敗。

## 2. plate lookup 輸入 contract

未來建議 helper：

```js
getQimenPlate({
  dunType,
  ju,
  hourPillar,
})
```

輸入欄位：

| 欄位 | 型別 | 必要 | 說明 |
| -- | -- | -- | -- |
| `dunType` | string | 是 | `yang` 或 `yin` |
| `ju` | number | 是 | 1～9 |
| `hourPillar` | string | 是 | 兩字干支時柱 |

來源：

* `dunType` 來自 formatter result。
* `ju` 來自 formatter result。
* `hourPillar` 來自 formatter result。
* 若有 manual override，`dunType` / `ju` 可改用 manual effective value。
* `hourPillar` 仍使用 formatter result，不由 manual override 改寫。

valid input 定義：

* `dunType` 必須是 `yang` 或 `yin`。
* `ju` 必須是 1～9 整數。
* `hourPillar` 必須是 60 甲子之一。
* helper 不負責計算時柱，只負責 lookup。

## 3. plate lookup 輸出 contract

建議回傳格式：

```js
{
  found: false,
  status: "nullPlate",
  message: "盤面資料尚未建立，目前僅顯示定局結果。",
  meta: {
    dunType: "yang",
    dunName: "陽遁",
    ju: 1,
    plateCount: 60
  },
  plate: null,
  lookup: {
    filePath: "data/qimen/plates/yang/ju-1.json",
    hourPillar: "甲子"
  }
}
```

欄位定義：

| 欄位 | 型別 | 說明 |
| -- | -- | -- |
| `found` | boolean | 是否取得可用盤面 |
| `status` | string | lookup 狀態 |
| `message` | string | UI fallback message |
| `meta` | object / null | plate file meta |
| `plate` | object / null | 盤面資料 |
| `lookup` | object | lookup metadata |

`status` 建議值：

| status | 說明 | UI 行為 |
| -- | -- | -- |
| `found` | 找到可用盤面 object | 顯示盤面 |
| `nullPlate` | key 存在但值為 null | 顯示缺盤 fallback |
| `missingPlate` | key 不存在 | 顯示缺盤 fallback |
| `missingFile` | plate file 不存在 | 顯示缺盤 fallback |
| `invalidInput` | 輸入不合法 | 顯示錯誤 fallback |
| `loadError` | JSON 載入或解析錯誤 | 顯示錯誤 fallback |

第一版可先用 `found: false` + status 描述狀態。`nullPlate` 是目前最常見狀態。`invalidInput` 不應 throw 到 UI，應轉成可顯示 fallback；但測試 helper 可視情況 assert status。

## 4. fallback message

### 4.1 nullPlate / missingPlate / missingFile

```text
盤面資料尚未建立，目前僅顯示定局結果。
```

### 4.2 invalidInput

```text
奇門盤面查詢參數不完整，暫時無法顯示盤面。
```

### 4.3 loadError

```text
奇門盤面資料讀取失敗，暫時無法顯示盤面。
```

UI 第一版可以統一顯示 generic fallback。debug / console 可保留更詳細的 status。缺盤 fallback 不應影響定局摘要顯示。

## 5. 與 formatter 的整合邊界

流程：

```text
resolveQimenJuFromFullTermCycleDraft(dateTimeText)
-> 得到 dunType / dunName / ju / hourPillar
-> getQimenPlate({ dunType, ju, hourPillar })
-> 有盤面則顯示盤面
-> 無盤面則顯示 fallback
```

整合邊界：

* formatter 不知道盤面是否存在。
* plate lookup 不重新計算遁別、局數、時柱。
* plate lookup 不修改 formatter result。
* formatter 成功但 plate lookup 失敗時，UI 仍應顯示定局摘要。
* plate lookup 層可以接收 manual effective dun / ju，但不得回寫 formatter。

## 6. 手動覆寫整合邊界

未來手動覆寫時，自動 formatter result：

```js
{
  dunType: "yang",
  dunName: "陽遁",
  ju: 1,
  hourPillar: "丙午"
}
```

manual override：

```js
{
  enabled: true,
  dunType: "yin",
  dunName: "陰遁",
  ju: 9
}
```

effective plate lookup input：

```js
{
  dunType: "yin",
  ju: 9,
  hourPillar: "丙午"
}
```

manual override 只影響 plate lookup 使用的 dunType / ju。hourPillar 仍由 formatter 計算，自動定局資訊仍保留顯示。不可把 manual override 當成 method switch，也不要回寫 resolver result。
