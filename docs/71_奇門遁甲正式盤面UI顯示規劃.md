# 奇門遁甲正式盤面 UI 顯示規劃

## 1. 本包目的

本文件是新第 114 包 docs-only 規劃文件，目的為規劃如何把 `getQimenPlate` 回傳的正式奇門盤面顯示到 UI。

本包只規劃 UI 顯示正式奇門盤面：

* 不修改 code。
* 不修改 tests。
* 不修改 data。
* 不接 UI。
* 不改 resolver。
* 不改 `getQimenPlate`。

## 2. 目前資料狀態

目前正式資料狀態：

* `data/qimen/plates/**` 已有 1080 formal plate objects。
* `getQimenPlate` 可回傳 `found` plate。
* plate object 包含 `zhiFuStar` / `zhiShiDoor` / `palaces`。
* UI 目前尚未顯示正式盤面。
* 目前只做傳統置閏法主線，無 method switch。

## 3. UI 接入資料流規劃

未來 UI 接入資料流建議：

```text
resolver 取得 dunType / ju / hourPillar
-> getQimenPlate({ dunType, ju, hourPillar })
-> found true：顯示正式盤面
-> invalidInput：顯示既有 fallback
-> missing / nullPlate：顯示盤面資料尚未建立 fallback
```

規劃原則：

* 使用目前 resolver 取得 `dunType` / `ju` / `hourPillar`。
* 呼叫 `getQimenPlate({ dunType, ju, hourPillar })`。
* 若 `found === true`，顯示正式盤面。
* 若 `invalidInput`，顯示既有 fallback。
* 若未來出現 missing / `nullPlate`，顯示「盤面資料尚未建立，目前僅顯示定局結果。」類 fallback。
* 不讓 UI 直接讀 JSON 檔。
* 不讓 UI 使用 preview / candidate output。

## 4. UI 顯示區塊位置規劃

依現有頁面結構，建議在奇門遁甲結果區中新增「奇門盤面」子區塊：

* 放在定局摘要之後。
* 不影響干支曆、九宮飛星、金函玉鏡、每日資訊等既有區塊。
* 董公擇日目前仍維持隱藏，不在本包處理。
* 建議第一版可用 collapsible / details 區塊，避免頁面太長。
* 若目前沒有 collapsible pattern，先用普通 section，但設計上預留可收合。

現有 `src/main.js` 已有奇門 section、summary、plate usage、manual override 與 fallback。第一版 UI 可沿用該區塊，新增盤面容器，不另開新的頂層頁面區塊。

## 5. 九宮 layout 規劃

九宮格建議用 3x3 grid 顯示，順序如下：

| row | palace order |
|---|---|
| 上排 | 巽、離、坤 |
| 中排 | 震、中、兌 |
| 下排 | 艮、坎、乾 |

每格顯示：

* 宮名 / 方位 / 洛書數，例如「坎｜北｜1」。
* 天盤干 `heavenStem`。
* 地盤干 `earthStem`。
* 九星 `star`。
* 八門 `door`。
* 八神 `deity`。
* 直符標記：若 `isZhiFuPalace === true`。
* 直使標記：若 `isZhiShiPalace === true`。
* `notes` 若有，第一版建議小字顯示；若資訊太多，可改為 debug 顯示。

## 6. 盤面摘要欄位

九宮格上方建議顯示盤面摘要：

* 遁別：陽遁 / 陰遁。
* 局數：一～九局。
* 時柱 `hourPillar`。
* 直符星 `zhiFuStar`。
* 直使門 `zhiShiDoor`。
* 旬首 `xunShou`：目前 null，第一版可不顯示或顯示「旬首：待補」。
* 資料來源：可顯示「盤面來源：1080.md」，或只放 debug。

建議第一版正式 UI 顯示遁別、局數、時柱、直符星、直使門；`xunShou` 與資料來源先保守處理，避免讓使用者誤以為旬首已推導完成。

## 7. center / 天禽顯示策略

center / 天禽顯示策略：

* 中宮一定顯示。
* 中宮 `star` 通常為天禽。
* `center.door` / `center.deity` 可能為 null。
* null 欄位 UI 顯示為「—」，不要顯示 `null` 字樣。
* 天禽寄宮目前未推導。
* 若 `zhiFuStar === "天禽"` 且 `center.isZhiFuPalace === true`，可在中宮顯示「直符」標記。
* 額外提示可寫：「天禽寄宮未推導，暫以中宮標示。」
* 第一版不要自行推導天禽寄宮。

## 8. 直符 / 直使標示策略

直符 / 直使標示策略：

* 宮格內顯示 badge：「直符」「直使」。
* 若同宮同時為直符與直使，兩個 badge 都顯示。
* badge 只根據 `plate.palaces[palace].isZhiFuPalace` / `isZhiShiPalace`。
* 不在 UI 重新判斷 star / door matching。
* 若資料缺 flag，fallback 不顯示 badge，不 throw。

## 9. null / missing fallback 策略

fallback 策略：

| case | UI behavior |
|---|---|
| `plate found false` | 顯示「盤面資料尚未建立，目前僅顯示定局結果。」 |
| `invalidInput` | 顯示「奇門盤面查詢參數不完整，暫時無法顯示盤面。」 |
| palace 欄位 null | 顯示「—」 |
| palace missing | 該格顯示「資料缺漏」並不中斷整頁 |
| `source.raw` | 正式 UI 不顯示，除非 debug mode |
| `notes` | 第一版建議小字顯示；若造成版面擁擠，改為 debug |

第一版建議 `notes` 先顯示在小字，因為 `notes` 是正式 plate object 的一部分；若後續發現大量資訊干擾閱讀，再改為 debug / collapsible。

## 10. 手機版 / 桌面版顯示密度

桌面版與手機版方案比較：

| option | 優點 | 風險 |
|---|---|---|
| 手機版仍維持 3x3 grid | 與九宮視覺一致，使用者容易對照方位 | 每格文字密度高，窄螢幕可能擁擠 |
| 手機版改為單欄 list | 可讀性高，欄位不易爆版 | 九宮相對位置感變弱 |

建議第一版：

* 桌面版維持 3x3 grid。
* 手機版仍保留 3x3 grid。
* 手機版縮小字級、欄位分行、避免太多說明字。
* 若太擠，再第 2 階段改成 mobile list。

## 12. 渲染函式規劃

未來可能新增 / 修改的 function：

| function | purpose |
|---|---|
| `renderQimenPlate(result)` | 根據 `getQimenPlate` 結果決定顯示盤面或 fallback |
| `renderQimenPlateSummary(plateResult)` | 渲染遁別、局數、時柱、直符星、直使門等摘要 |
| `renderQimenPlateGrid(plate)` | 渲染 3x3 九宮格 |
| `formatQimenPalaceCell(palace)` | 將單宮資料轉為 DOM / display rows |
| `formatNullableQimenValue(value)` | 將 null / undefined / 空字串統一顯示為「—」 |
| `getQimenPlateDisplayOrder()` | 回傳巽、離、坤、震、中、兌、艮、坎、乾順序 |

放置位置取捨：

* 第一版建議先放 `src/main.js`，因為現有奇門 UI 互動、summary、fallback 都集中在 `src/main.js`。
* 若渲染函式變大，或需要單獨測試 DOM helper，再拆成獨立 UI helper。
* 第一版不應改 `getQimenPlate`，也不應讓 UI helper 直接 import JSON。

## 13. 測試規劃

未來 UI 實作包應測：

* `node --check src/main.js`。
* `npm test`。
* `getQimenPlate` found 不變。
* render function 對 found plate 不 throw。
* null 欄位顯示為「—」。
* center 顯示天禽。
* 直符 / 直使 badge 顯示。
* `invalidInput` / missing fallback。
* `git diff --check`。
* 若現有測試不含 DOM，可先以 `node --check` + manual preview 驗收。

若要補 DOM 測試，建議先評估現有測試架構是否已支援瀏覽器或 DOM mock；不要在 UI 第一包同時引入大型測試框架。

## 14. 分包建議

建議後續分包：

| package | scope |
|---|---|
| 第 115 包 | UI DOM 容器與 placeholder，不接資料，最小 HTML / CSS |
| 第 116 包 | `main.js` 接 `getQimenPlate` 並渲染盤面 summary / grid |
| 第 117 包 | CSS 細修與手機版顯示 |
| 第 118 包 | UI 驗收報告 docs-only |

這樣分包可避免一包同時改 DOM、資料接線、CSS 與驗收，降低 UI 迴歸風險。
