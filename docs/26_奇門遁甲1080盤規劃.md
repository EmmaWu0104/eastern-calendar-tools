# 奇門遁甲 1080 盤規劃

本文件只記錄奇門遁甲 1080 盤功能的整體規劃，不代表已實作。本包只新增規格文件，不修改程式碼、UI、測試或資料 JSON。

## 一、功能目標

奇門遁甲功能最終目標，是讓使用者輸入查詢時間後，可以取得：

1. 實際節氣。
2. 奇門節氣。
3. 狀態：正常、超神、接氣、置閏或置閏後接氣。
4. 上元 / 中元 / 下元。
5. 陰遁 / 陽遁。
6. 局數。
7. 時辰干支。
8. 對應的完整奇門盤面。

第一階段先只處理奇門定局，也就是實際節氣、奇門節氣、狀態、上中下元、陰陽遁、局數與時辰干支。

完整 1080 盤面資料不在第一階段處理。

## 二、本階段範圍

本階段只規劃奇門遁甲第一包，範圍如下：

- 建立奇門遁甲 1080 盤整體工程方向。
- 明確區分「定局 resolver」與「1080 盤 JSON 查表」。
- 規劃 1080 盤資料檔拆分方式。
- 規劃盤面 JSON 可逐步補齊。
- 規劃 UI 在有盤面與無盤面時的顯示原則。
- 建立後續分包順序。

第一階段實作時，應優先完成：

- 實際節氣判斷。
- 奇門節氣判斷。
- 超神 / 接氣 / 置閏 / 置閏後接氣狀態。
- 上元 / 中元 / 下元。
- 陰陽遁。
- 局數。
- 查詢時辰干支。

## 三、非本階段範圍

本階段不處理：

- 完整 1080 盤面資料。
- 九宮內天盤、地盤、人盤、神盤、星、門、奇儀等完整欄位。
- 盤面資料人工校對。
- 盤面 JSON 完整性測試。
- UI 完整盤面呈現。
- 盤面互動功能。
- 真太陽時。
- 拆補法、茅山法或無閏簡化法。
- 由程式直接推演完整奇門盤。

## 四、整體架構分層

奇門遁甲功能建議分為四層：

### 1. 時間與干支層

沿用本工具既有干支曆核心：

- 使用 Asia/Taipei 手錶時間。
- 不使用真太陽時。
- 日柱採 23:00 換日。
- 時辰依查詢時間取得。
- 節氣使用既有 24 節氣資料與實際交節時間。

### 2. 定局 resolver

定局 resolver 負責把查詢時間解析成奇門定局結果。

輸入可包含：

- 查詢時間。
- 實際節氣資料。
- 日柱與時柱。

輸出應包含：

```json
{
  "actualSolarTerm": "芒種",
  "qimenSolarTerm": "夏至",
  "status": "接氣",
  "yuan": "上元",
  "dunType": "陰遁",
  "ju": 9,
  "hourPillar": "甲子"
}
```

定局 resolver 是奇門節氣、超神接氣、置閏、陰陽遁與局數判斷的唯一責任層。

### 3. 1080 盤 JSON 查表層

1080 盤之後採 JSON 查表，不由程式直接推盤。

查表 key 建議由定局 resolver 的輸出組成：

- `dunType`：`yang` / `yin`
- `ju`：`1` 到 `9`
- `hourPillar`：60 甲子時柱

查表層只根據 key 讀取資料，不負責判斷規則。

### 4. UI 顯示層

UI 顯示層負責呈現：

- 定局摘要。
- 實際節氣與奇門節氣。
- 狀態。
- 陰陽遁局數。
- 時辰干支。
- 若查得到盤面 JSON，才顯示完整盤面。

UI 不應直接判斷超神、接氣、置閏或局數。

## 五、定局 resolver 與 1080 盤 JSON 查表分離

定局 resolver 與盤面 JSON 查表必須分離。

定局 resolver 負責：

- 實際節氣。
- 奇門節氣。
- 正常 / 超神 / 接氣 / 置閏 / 置閏後接氣。
- 上元 / 中元 / 下元。
- 陰陽遁。
- 局數。
- 時辰干支。

1080 盤 JSON 負責：

- 指定陰陽遁。
- 指定局數。
- 指定時辰干支。
- 對應的完整盤面內容。

盤面 JSON 不負責判斷超神、接氣、置閏、實際節氣或奇門節氣；這些由定局 resolver 負責。

這樣可以避免把規則判斷散落在資料檔內，也能讓定局規則先穩定，再逐步補齊盤面資料。

## 六、1080 盤資料規劃

奇門 1080 盤可拆為：

```text
陽遁 9 局 x 60 時辰盤 = 540 盤
陰遁 9 局 x 60 時辰盤 = 540 盤
合計 1080 盤
```

1080 盤 JSON 建議拆為 18 檔：

```text
data/qimen/plates/yang/ju-1.json
data/qimen/plates/yang/ju-2.json
data/qimen/plates/yang/ju-3.json
data/qimen/plates/yang/ju-4.json
data/qimen/plates/yang/ju-5.json
data/qimen/plates/yang/ju-6.json
data/qimen/plates/yang/ju-7.json
data/qimen/plates/yang/ju-8.json
data/qimen/plates/yang/ju-9.json

data/qimen/plates/yin/ju-1.json
data/qimen/plates/yin/ju-2.json
data/qimen/plates/yin/ju-3.json
data/qimen/plates/yin/ju-4.json
data/qimen/plates/yin/ju-5.json
data/qimen/plates/yin/ju-6.json
data/qimen/plates/yin/ju-7.json
data/qimen/plates/yin/ju-8.json
data/qimen/plates/yin/ju-9.json
```

每檔包含 60 個時辰盤。

單檔資料結構可先規劃為：

```json
{
  "meta": {
    "dunType": "yang",
    "ju": 1,
    "count": 60
  },
  "plates": {
    "甲子": null,
    "乙丑": null,
    "丙寅": null
  }
}
```

後續盤面補齊後，可逐筆由 `null` 改為完整盤面物件。

## 七、盤面 JSON 漸進式補齊策略

1080 盤資料可逐步補齊。

第一階段不要求建立完整 1080 盤面資料，也不要求補齊任一完整盤面。

後續資料補齊建議：

1. 先建立 18 檔檔案骨架。
2. 每檔先列出 60 甲子時辰 key。
3. 尚未建立的盤面填 `null`。
4. 每次只補一小批已校對盤面。
5. 補資料時同步加入資料完整性測試。
6. UI 遇到 `null` 時不得當作錯誤。

若查得到盤面 JSON，才顯示完整盤面。

若查不到盤面 JSON 或該時辰資料為 `null`，仍應正常顯示陰陽遁局數，並顯示：

```text
盤面資料尚未建立，目前僅顯示定局結果
```

## 八、UI 顯示原則

奇門遁甲 UI 第一階段應以定局摘要為主。

基本摘要建議顯示：

```text
奇門遁甲
實際節氣：芒種
奇門節氣：夏至
狀態：接氣
元：上元
陰陽遁：陰遁
局數：九局
時辰：甲子時
```

若實際節氣與奇門節氣相同，可顯示狀態為 `正常`。

若實際節氣與奇門節氣不同，應清楚顯示狀態，例如：

- 超神。
- 接氣。
- 置閏。
- 置閏後接氣。

完整盤面顯示原則：

- 查得到盤面 JSON，才顯示完整盤面。
- 查不到盤面 JSON 時，不影響定局摘要。
- 該時辰盤面為 `null` 時，不顯示空白九宮格。
- 未建資料時顯示固定提示文字。
- UI 不應讓使用者誤以為盤面已完成。
