# 奇門盤面標記規則規劃

## 1. 本包目的

本文件是新第 121 包 docs-only 規劃文件，目的為整理奇門盤面標記規則、資料責任邊界、UI 顯示位置、helper 設計、測試案例與後續分包。

本包只規劃盤面標記規則：

* 不修改 code。
* 不修改 tests。
* 不修改 data。
* 不產生 JSON。
* 不接 UI。
* 不做新推導實作。

## 2. 規則分類

本階段標記規則分為三類：

| 類別 | 推導來源 |
|---|---|
| 天盤干刑 / 墓 / 制 | 由 palace key + `heavenStem` 推導 |
| 八門門迫 | 由 palace key + `door` 推導 |
| 中宮寄宮 | 由 center `heavenStem` / `earthStem` 與天芮所在宮推導 |

## 3. 不放入 data/1080.md 的原因

不建議把本階段標記直接寫入 `data/1080.md`：

* 刑 / 墓 / 制 是固定規則，可由 palace + `heavenStem` 推導。
* 門迫是固定規則，可由 palace + `door` 推導。
* 中宮寄宮可由 center stems 與天芮所在宮推導。
* 這些屬於衍生標記，不是 `1080.md` 原始盤面文字本體。
* 因此不應手工塞進 `data/1080.md`，也不應直接手修 `data/qimen/plates/**`。

若未來發現原始盤面內容本身錯誤，仍應先修 `data/1080.md`，再重跑 parser / converter / adapter / writer。

## 6. 中宮寄宮規則

中宮寄宮規則：

* 中宮地盤干 `center.earthStem` 寄到坤宮。
* 顯示位置：坤宮地盤干右邊。
* 中宮天盤干 `center.heavenStem` 寄到「天芮」所在宮。
* 天芮所在宮判斷：尋找 `plate.palaces[palaceKey].star === "天芮"`。
* 顯示位置：該宮天盤干右邊。
* 中宮本身 `heavenStem` / `earthStem` 仍保留顯示。
* 若該宮天盤干已有刑 / 墓 / 制 標記，中宮天盤寄宮標記再往右放。
* 若找不到天芮所在宮，不 throw，僅不顯示中宮天盤寄宮，並可回傳 warning diagnostics。
* 若 center `heavenStem` / `earthStem` 為 null，不顯示寄宮標記。

## 7. UI 顯示位置規劃

UI 顯示位置建議：

* 天盤干右側顯示小方框「刑 / 墓 / 制」。
* 八門圈圈右下角顯示小方框「迫」。
* 坤宮地盤干右側顯示中宮地盤寄宮干，建議用小字或小框。
* 天芮所在宮天盤干右側顯示中宮天盤寄宮干，若已有刑墓制則再往右。
* 不改現有九宮主要布局。
* 不影響直符 / 直使 badge。
* null fallback 仍顯示「—」。

## 8. helper 設計建議

建議未來新增：

```text
src/qimenPlateMarkers.js
```

建議 exports：

```js
getQimenHeavenStemMarker(palaceKey, heavenStem)
getQimenDoorPoMarker(palaceKey, door)
findQimenTianRuiPalaceKey(plate)
getQimenCenterStemPlacements(plate)
decorateQimenPlateMarkers(plate)
```

設計原則：

* helper 不修改原 plate object。
* 回傳新的 markers object 或 decorated clone。
* UI 只讀 markers，不自行推導規則。
* helper 集中規則，方便測試。

## 9. markers object 草案

markers object 草案：

```json
{
  "palaces": {
    "kan": {
      "heavenStemMarker": "制",
      "doorPo": null,
      "centerHeavenStem": null,
      "centerEarthStem": null
    },
    "kun": {
      "heavenStemMarker": "刑",
      "doorPo": "迫",
      "centerHeavenStem": null,
      "centerEarthStem": "戊"
    }
  },
  "diagnostics": []
}
```

命名可在實作時微調，但需維持幾個原則：

* palace key 對應正式 plate `palaces` key。
* 沒有命中的標記回傳 null。
* diagnostics 可記錄找不到天芮、缺 center、缺 palaces 等非致命狀況。

## 10. 測試案例規劃

未來 helper 測試至少包含：

* 坎宮 `heavenStem` 丙 -> 制。
* 坎宮 `heavenStem` 丁 -> 制。
* 艮宮 `heavenStem` 庚 -> 刑。
* 乾宮 `heavenStem` 丙 -> 墓。
* 離宮 `door` 休 -> 迫。
* 坤宮 `door` 傷 -> 迫。
* 兌宮 `door` 景 -> 迫。
* 無規則組合 -> null。
* center `earthStem` 寄到 `kun`。
* center `heavenStem` 寄到天芮所在宮。
* 找不到天芮不 throw。
* helper 不污染原 plate object。
