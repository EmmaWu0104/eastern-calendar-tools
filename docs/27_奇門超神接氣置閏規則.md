# 奇門超神接氣置閏規則

本文件整理奇門遁甲置閏法規格，作為後續奇門定局 resolver 的依據。本文件只記錄規則與工程邊界，不代表已實作。

## 一、規則總原則

第一版奇門遁甲定局採以下總原則：

1. 本工具只採奇門遁甲置閏法。
2. 不支援拆補法。
3. 不支援茅山法。
4. 不支援無閏簡化法。
5. 不預留 `method` 參數。

奇門定局 resolver 應直接以置閏法為唯一規則來源，不在 UI 或 API 層提供流派切換。

## 二、時間基準

第一版時間基準如下：

- 第一版仍採 Asia/Taipei 手錶時間。
- 第一版不使用真太陽時。
- 日柱採 23:00 換日。
- 節氣使用實際交節時間。

節氣交節時間用於判斷實際節氣；日柱與符頭序列用於推進奇門節氣局。兩者需要同時保留，不能互相覆蓋。

## 三、符頭定義

符頭用於推進奇門三元序列。

規則：

- 符頭一律是日干為甲或己。
- 符頭不是任意旬頭。
- 判斷符頭時，日柱沿用本工具 23:00 換日規則。

符頭只負責標記三元推進節點，不直接等同於實際節氣交節時間。

## 四、上中下元判定

上中下元由符頭日柱地支判斷：

- 上元符頭地支：子、午、卯、酉。
- 中元符頭地支：寅、申、巳、亥。
- 下元符頭地支：辰、戌、丑、未。

每一節氣理想上有上元 5 日、中元 5 日、下元 5 日。

三元推進應以符頭開始，連續 5 日為一元。實作時需注意日柱 23:00 換日，因此三元日界線應與日柱有效日一致。

## 五、置閏法推進邏輯

置閏法以符頭三元序列推進奇門節氣局。

基本概念：

- 實際節氣由天文交節時間決定。
- 奇門節氣由符頭三元序列推進。
- 正常情況下，一個奇門節氣走上元、中元、下元，共 15 日。
- 若符頭三元序列與實際節氣出現前後差，會形成超神或接氣。
- 若超神達九日以上，且進入指定置閏窗口，才可置閏。

接氣不作為固定長度規則。

接氣是置閏法推進過程中自然形成的狀態。

因此 resolver 不應用「固定提前幾日就是接氣」這種簡化判斷，而應由奇門節氣局序與實際節氣比對後得到狀態。

## 六、超神九日置閏

超神表示奇門節氣局已走在實際節氣之前。

規則：

- 超神九日後，僅能在芒種或大雪後置閏。
- 不在其他節氣任意置閏。
- 未達九日不置閏。
- 達九日以上也必須等待芒種或大雪窗口。

超神天數應依奇門節氣局已超前實際節氣的三元日數計算。後續實作需以人工驗證案例校對，不可只以節氣交節日差粗略推算。

## 七、芒種 / 大雪置閏窗口

置閏只在芒種與大雪兩個窗口處理。

芒種置閏時：

- 重走芒種上中下三元，共 15 日。
- 置閏期間奇門節氣仍顯示為芒種。
- 置閏結束後才進入後續奇門節氣局。

大雪置閏時：

- 重走大雪上中下三元，共 15 日。
- 置閏期間奇門節氣仍顯示為大雪。
- 置閏結束後才進入後續奇門節氣局。

UI 狀態應能區分：

- 置閏。
- 置閏後接氣。

## 八、陰陽遁切換原則

陰陽遁切換以奇門節氣局序為準，不以實際夏至 / 冬至交節時刻直接切換。

規則：

- 進入奇門夏至上元時，開始陰遁。
- 進入奇門冬至上元時，開始陽遁。

因此可能出現：

- 實際夏至未到，但奇門局序已進入夏至上元，提前開始陰遁。
- 實際冬至已到，但奇門局序仍在置閏大雪，尚未開始陽遁。

UI 顯示陰陽遁時，必須依奇門節氣局序結果，不可直接依實際夏至 / 冬至交節時間判斷。

## 十、實際節氣與奇門節氣顯示原則

UI 必須同時顯示實際節氣與奇門節氣。

若兩者相同，可顯示：

```text
實際節氣：芒種
奇門節氣：芒種
狀態：正授（僅限交節當天的奇門有效日為符頭）
```

若兩者不同，應顯示狀態：

- 超神。
- 接氣。
- 置閏。
- 置閏後接氣。

顯示範例：

```text
實際節氣：芒種
奇門節氣：夏至
狀態：接氣
陰陽遁：陰遁
局數：九局
```

狀態由定局 resolver 回傳，UI 不自行推導。

## 十一、2027 年芒種驗證案例

案例一：2027 芒種

- 2027 芒種交節時間：2027/6/6 05:26。
- 但 2027/6/6 已是芒種中元第三天。
- 超神天數 = 芒種上元 5 日 + 芒種中元 3 日 = 8 日。
- 未達九日，不置閏。
- 2027/6/13 芒種下元結束。
- 2027/6/14 直接進入奇門夏至上元。
- 雖然實際夏至在 2027/6/21 22:11，仍以奇門局序為準，2027/6/14 開始陰遁。

此案例驗證重點：

- 未達九日不可置閏。
- 不因芒種窗口存在就一定置閏。
- 陰陽遁切換依奇門夏至上元，不依實際夏至交節時刻。

## 十二、2027 年大雪驗證案例

案例二：2027 大雪

- 2027 大雪交節時間：2027/12/7 16:38。
- 但該日已是大雪下元第二天。
- 超神天數 = 大雪上元 5 日 + 大雪中元 5 日 + 大雪下元 2 日 = 12 日。
- 已達九日以上，因此在大雪後置閏。
- 2027/12/11 起重新走一次大雪三元，共 15 日。
- 即使實際冬至在 2027/12/22 10:42，奇門局序仍在置閏大雪中。
- 2027/12/25 置閏大雪下元結束。
- 2027/12/26 才進入奇門冬至上元，並開始陽遁。

此案例驗證重點：

- 超神達九日以上後，在大雪窗口置閏。
- 置閏重走大雪上中下三元，共 15 日。
- 實際冬至已到時，不直接切陽遁。
- 進入奇門冬至上元時才開始陽遁。

## 十三、後續實作注意事項

後續實作定局 resolver 時，應注意：

- 規則只採置閏法，不做 method 分支。
- 先建立可人工驗證的三元推進資料，再接 UI。
- 實際節氣與奇門節氣都要保留。
- 狀態不可只用日期差推測，應由奇門局序與實際節氣比對取得。
- 接氣不可寫成固定長度規則。
- 超神九日後也只能在芒種或大雪後置閏。
- 陰陽遁不可直接用實際夏至 / 冬至切換。
- 日柱與三元日界線需沿用 23:00 換日。
- 節氣交節時間仍使用實際交節時間。
- 第一版不引入真太陽時。
- 不修改既有 24 節氣 JSON。
- 不修改既有干支曆核心規則，除非另開專包並有測試。

建議 resolver 回傳格式：

```json
{
  "actualSolarTerm": "大雪",
  "qimenSolarTerm": "大雪",
  "status": "置閏",
  "yuan": "下元",
  "dunType": "陰遁",
  "ju": 1,
  "hourPillar": "甲子",
  "notes": [
    "奇門局序仍在置閏大雪中，尚未進入冬至上元。"
  ]
}
```
