# 董公擇日資料結構與整理規則

## 1. 目前完成狀態

- 已完成董公擇日第一版骨架。
- 已新增 `data/dong_gong_day_selection.json`。
- 已新增 `src/dongGongDaySelection.js`。
- 已在 UI 顯示「董公擇日」卡片。
- UI 位置在四柱 / 節氣資訊下方、九宮飛星上方。
- 第一版目前只放 3 筆樣本資料，用於驗證流程。
- 查無資料顯示「資料待補」。

## 2. 定位與範圍

董公擇日第一版定位為「日期層級查表」。本功能只呈現目前日期在董公條文中的日期層級說明，不做完整專業擇日裁決。

- 使用節氣月令，不使用農曆月。
- 使用既有干支曆邏輯取得的節氣月支。
- 使用既有干支曆邏輯取得的日柱。
- 使用既有建除十二神結果，不重寫建除公式。
- 不做自動總分。
- 不合本命。
- 不合山向。
- 不合時辰。
- 不做安葬、嫁娶、修造等完整專業擇日裁決。
- UI 固定提醒：「董公擇日僅列日期層級，未合本命、山向、時辰。」

## 3. 明確排除項目

- 本階段不加入金神七煞。
- 本階段不加入二十八宿。
- 本階段不加入玉匣記九星值日。
- 本階段不加入煞貢、直星、人專的獨立推算。
- 本階段不加入其他通書神煞。
- 若原文條文中提到某些神煞，只可作為董公條文內的文字、`stars` 或 `notes` 保存，不可新增外部推算規則。
- 未來若要加入上述系統，必須另開獨立模組與文件。

## 4. 資料來源原則

- 目前董公資料應以《董公選擇日要覽》系統為主。
- 網路來源多為轉載，可能有異體字、簡繁轉換、錯字、重複段落、標點差異。
- 後續整理資料時，應先選定一個主本，其餘來源只作校對。
- 若來源互相衝突，資料檔應保留 `notes`，不能自行武斷合併。
- 不要因為其他通書或文章說法而覆寫董公條文本身。

## 5. 核心查詢規則

目前查詢流程如下：

1. 由既有干支曆取得節氣月支，例如「寅」。
2. 由既有干支曆取得日柱，例如「丁酉」。
3. 由既有建除功能取得建除十二神，例如「危」。
4. 以 `monthBranch + dayBranch` 查主資料。
5. 若 `stemBranchOverrides[dayPillar]` 存在，使用完整日柱覆寫。
6. 若無覆寫，使用 general 資料。
7. 若查無資料，回傳 fallback，不中斷 UI。

注意：建除十二神在此是既有資料結果與條文索引輔助，不代表直接採用建除本身吉凶來判斷董公擇日。

## 6. 資料檔欄位說明

`data/dong_gong_day_selection.json` 目前採陣列格式，每一筆代表一個「節氣月支 + 日支」主條文。

- `monthBranch`：節氣月支，例如 `寅`、`卯`。必須使用既有干支曆節氣月令，不使用農曆月。
- `monthName`：該月令的人類可讀名稱，例如 `正月`。只作顯示與整理輔助。
- `jianChu`：該月支與日支對應的建除十二神名稱，例如 `危`。使用既有建除計算結果校對，不新增公式。
- `dayBranch`：日柱地支，例如 `酉`。
- `generalLevel`：主條文層級，例如 `凶`、`分干支`。若完整日柱有覆寫，顯示時以覆寫為優先。
- `title`：條文標題，例如 `正月危酉日`。
- `summary`：主條文摘要，應保留原文核心語意，不自行擴張解釋。
- `suitable`：宜事項陣列。
- `avoid`：忌事項陣列。
- `stars`：條文中明確提到的神煞、吉星或凶星名稱，只作顯示。
- `notes`：條件、限制、校勘疑問、來源差異、方位條件、本命條件等補充說明。
- `stemBranchOverrides`：完整日柱覆寫資料。key 為完整日柱，例如 `丁酉`、`辛酉`。

`stemBranchOverrides` 內每個完整日柱可使用以下欄位：

- `level`：該完整日柱的覆寫層級。
- `summary`：該完整日柱的覆寫摘要。
- `suitable`：該完整日柱的宜事項。
- `avoid`：該完整日柱的忌事項。
- `stars`：該完整日柱條文中明確提到的星曜或神煞。
- `notes`：該完整日柱的限制、條件或來源備註。

## 7. level 使用規則

目前建議可用值：

- `大吉`
- `吉`
- `次吉`
- `平`
- `慎用`
- `凶`
- `分干支`
- `資料待補`

使用原則：

- 原文如果明確說大吉，才用 `大吉`。
- 原文只是可用、宜用，優先用 `吉` 或 `次吉`。
- 原文有吉有凶、需看事情類別，優先用 `慎用` 或 `分干支`。
- 不應自行用現代主觀判斷加分或扣分。
- 不做數值化分數。

## 8. 宜忌整理規則

- `suitable` 放「宜」事項。
- `avoid` 放「忌」事項。
- 同一日可能同時有宜與忌。
- 同一日可能只適合某些事，不代表所有事皆吉。
- 若原文說「百事不宜」，可在 `avoid` 放 `百事` 或 `用事`，並在 `summary` 保留原文語意。
- 若原文說「只不宜起造婚姻」，應明確放入 `avoid`，不可把整日視為全凶。
- 若有「小事可、大事不可」，先放 `notes`，不要自動判斷。

## 9. stars 與 notes 整理規則

- `stars` 儲存條文中明確提到的神煞、吉星或凶星名稱。
- `stars` 只作顯示，不作額外推算。
- `notes` 儲存條件、限制、校勘疑問、來源差異、方位條件、本命條件等。
- 例如「水土生人用之尤吉」應放 `notes`。
- 例如「乾、巽二宮起造皆吉」應放 `notes`。
- 例如「其餘酉日不可用」應放 `notes`。

## 10. 干支覆寫規則

- 只要原文對完整日柱有特別說明，就應放入 `stemBranchOverrides`。
- 覆寫資料優先於 general 資料。
- 覆寫不需要繼承 general 的宜忌，除非原文語意明確相同。
- 若同一日支中只有部分干支可用，應清楚標示其他干支不可用或資料未明。
- 不要把「某一干支可用」擴大成「整個日支皆可用」。

## 11. 後續補資料策略

- 不要一次補完 144 條。
- 建議每包補 1 個節氣月，也就是 12 條主資料。
- 每包補完後都要新增測試。
- 每包都要確認沒有引入金神七煞、二十八宿或其他外部規則。
- 每包完成後執行 `npm test`、`git diff --check`，必要時執行 `node --check`。

## 12. 測試建議

後續每月資料補齊時應測：

- 至少 1 筆一般主資料。
- 至少 1 筆完整日柱覆寫。
- 至少 1 筆查無資料 fallback。
- 若該月有「某干支可用、其餘不可用」條文，需測可用與不可用各一筆。
- 若該月有「宜某事、忌某事」並存條文，需測 `suitable` 與 `avoid`。
