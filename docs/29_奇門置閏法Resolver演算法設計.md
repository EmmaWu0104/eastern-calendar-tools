# 奇門置閏法 Resolver 演算法設計

本文件整理奇門遁甲置閏法 resolver 的演算法設計。本文只作為後續實作依據，不代表 resolver 已實作。

## 一、文件目的

本文件目的，是在實作奇門定局 resolver 前，先固定責任邊界、資料來源、推進模型與狀態判斷方式。

本工具只採置閏法：

- 不支援拆補法。
- 不支援茅山法。
- 不支援無閏簡化法。
- 不預留 `method` 參數。

後續 resolver 應直接以置閏法為唯一規則，不在 API、資料或 UI 層加入流派切換。

## 二、Resolver 責任範圍

resolver 的責任是把查詢時間解析為奇門定局結果。

resolver 應輸出：

- `actualSolarTerm`
- `qimenSolarTerm`
- `status`
- `yuan`
- `dunType`
- `dunName`
- `ju`
- `hourPillar`
- `isIntercalary`
- `notes`

resolver 應負責：

- 取得查詢時間所屬實際節氣。
- 建立或查找奇門三元局序 timeline。
- 判斷查詢時間落在哪個奇門節氣、元別與是否置閏。
- 由奇門節氣與元別查局數表。
- 依奇門節氣局序決定陰陽遁。
- 產生狀態：正授、超神、接氣、置閏或置閏後接氣。
- 回傳後續 1080 盤查表所需的定局 key。

## 三、Resolver 非責任範圍

resolver 不負責：

- 讀取完整 1080 盤面。
- 顯示 UI。
- 推演九宮盤。
- 直接產生天盤、地盤、人盤、神盤、星、門、奇儀等完整盤面。
- 修改既有干支曆核心。
- 修改既有 24 節氣 JSON。
- 修改 `data/qimen/plates/` 的盤面資料。
- 從 1080 盤 JSON 反推定局規則。

1080 盤 JSON 不參與定局判斷。它只在 resolver 已完成定局後，依 `dunType + ju + hourPillar` 進行查表。

## 四、輸入資料

resolver 建議輸入：

```js
resolveQimenJu(dateTime, calendarResult, solarTerms)
```

其中：

- `dateTime`：查詢時間，第一版使用 Asia/Taipei 手錶時間。
- `calendarResult`：既有干支曆結果，可提供日柱、時柱等資訊。
- `solarTerms`：既有 24 節氣實際交節時間資料。

若後續拆成更純的函式，可把日柱序列、時柱與節氣資料拆為獨立參數，但第一版不需要新增流派參數。

## 五、輸出資料

建議回傳格式：

```json
{
  "actualSolarTerm": "大雪",
  "qimenSolarTerm": "大雪",
  "status": "置閏",
  "yuan": "上元",
  "dunType": "yin",
  "dunName": "陰遁",
  "ju": 4,
  "hourPillar": "甲子",
  "isIntercalary": true,
  "notes": [
    "查詢時間落在閏大雪三元內。"
  ]
}
```

欄位說明：

- `actualSolarTerm`：查詢時間依實際交節時間得到的節氣。
- `qimenSolarTerm`：查詢時間落入奇門三元局序後得到的節氣。
- `status`：實際節氣與奇門節氣比對後的狀態。
- `yuan`：上元、中元或下元。
- `dunType`：`yang` 或 `yin`。
- `dunName`：陽遁或陰遁。
- `ju`：局數，1 到 9。
- `hourPillar`：查詢時辰干支。
- `isIntercalary`：是否落在閏芒種或閏大雪三元內。
- `notes`：補充說明，供 UI 或除錯顯示。

## 六、關鍵資料來源

resolver 需要的資料來源：

- 既有 24 節氣實際交節時間資料。
- 既有干支曆日柱與時柱邏輯。
- `data/qimen/qimen_yuan_ju_table.json`。
- 符頭與上中下元判定規則。

局數表使用 `data/qimen/qimen_yuan_ju_table.json`。

`data/qimen/plates/` 下的 1080 盤 JSON 不參與定局判斷。

## 七、時間基準

第一版時間基準如下：

- resolver 應使用既有 Asia/Taipei 手錶時間。
- resolver 第一版不使用真太陽時。
- 日柱與三元日界線沿用 23:00 換日。
- 節氣使用既有實際交節時間資料。

注意：

- 實際節氣以交節時間判斷。
- 三元日界線以 23:00 換日後的有效日判斷。
- 這兩套時間基準要同時存在，不可互相取代。

## 八、三元序列推進模型

演算法分為兩個概念層：

### A. 建立奇門三元局序 timeline

奇門局序不是單純依實際節氣切換。

奇門局序以符頭三元推進：

- 每個奇門節氣理想上有上元、中元、下元，各 5 日。
- 符頭一律為日干甲或己。
- 上元符頭地支為：子、午、卯、酉。
- 中元符頭地支為：寅、申、巳、亥。
- 下元符頭地支為：辰、戌、丑、未。
- 三元日界線應使用 23:00 換日後的有效日。

timeline 應由一段段區間組成，例如：

```json
{
  "qimenSolarTerm": "大雪",
  "yuan": "上元",
  "start": "2027-12-11T23:00:00+08:00",
  "end": "2027-12-16T23:00:00+08:00",
  "isIntercalary": true
}
```

若在芒種或大雪窗口達超神九日以上，應插入閏芒種或閏大雪三元。

置閏時重走該節氣上中下三元，共 15 日。

### B. 查詢時間落點解析

查詢時間落點解析流程：

1. 查詢時間先取得實際節氣 `actualSolarTerm`。
2. 查詢時間再落入奇門三元局序 timeline，取得 `qimenSolarTerm`、`yuan`、`isIntercalary`。
3. 由 `qimenSolarTerm + yuan` 查 `data/qimen/qimen_yuan_ju_table.json` 取得 `dunType` 與 `ju`。
4. `hourPillar` 由既有干支曆 / 時柱邏輯取得。
5. `status` 由 `actualSolarTerm` 與 `qimenSolarTerm`，以及是否置閏判斷。

## 九、實際節氣與奇門節氣比對模型

實際節氣與奇門節氣要分開保存：

- `actualSolarTerm`：天文交節後的實際節氣。
- `qimenSolarTerm`：符頭三元序列推進後的奇門節氣。

比對時應使用固定的 24 節氣順序。

建議節氣順序常數：

```js
[
  "冬至", "小寒", "大寒", "立春", "雨水", "驚蟄",
  "春分", "清明", "穀雨", "立夏", "小滿", "芒種",
  "夏至", "小暑", "大暑", "立秋", "處暑", "白露",
  "秋分", "寒露", "霜降", "立冬", "小雪", "大雪"
]
```

實作時需處理循環序列，避免冬至、大雪跨年比較出錯。

## 十、狀態判斷模型

狀態定義如下。

### 1. 正授

- 該奇門節氣的上元符頭有效日，與同名實際節氣的有效日相同。
- 符頭日為甲子、甲午、己卯或己酉，且不在置閏區間。
- 不可只以 `actualSolarTerm` 與 `qimenSolarTerm` 相同判定。

### 2. 超神

- `qimenSolarTerm` 走在 `actualSolarTerm` 之前。
- 且不在置閏區間。

例如實際仍為芒種，但奇門局序已進入夏至。

### 3. 接氣

- `actualSolarTerm` 走在 `qimenSolarTerm` 之前。
- 且不在置閏區間。

接氣不是固定天數規則。

不可以寫「接氣最多五日」。

接氣是奇門局序與實際節氣比對後自然得到的狀態。

### 4. 置閏

- 查詢時間落在閏芒種或閏大雪三元內。
- 且實際節氣尚未走到後續節氣太遠時可顯示置閏。
- 實作上可先以 `isIntercalary = true` 作為底層判斷。

### 5. 置閏後接氣

- 查詢時間仍落在閏芒種或閏大雪三元內。
- 但 `actualSolarTerm` 已經進入後續節氣，例如實際冬至已到，但奇門仍在閏大雪。
- UI 顯示可用「置閏後接氣」。
- 底層可用 `isIntercalary = true + actualSolarTerm 超前 qimenSolarTerm` 判斷。

狀態名稱可在實作前再精修，但 `qimenSolarTerm`、`yuan`、`dunType`、`ju`、`isIntercalary` 應優先作為硬驗證欄位。

## 十一、超神九日判斷

超神九日用於判斷是否需要在芒種或大雪窗口置閏。

規則：

- 超神九日後，僅能在芒種或大雪後置閏。
- 未達九日不置閏。
- 達九日以上也不能在其他節氣任意置閏。

判斷方式應以奇門三元局序相對於實際節氣的超前日數為準，不應只用兩個交節時間的日期差粗略推算。

2027 芒種案例：

- 2027/6/6 已是芒種中元第三天。
- 超神天數為 8 日。
- 未達九日，不置閏。

2027 大雪案例：

- 2027/12/7 已是大雪下元第二天。
- 超神天數為 12 日。
- 已達九日以上，因此在大雪後置閏。

## 十二、芒種 / 大雪置閏處理

置閏只在芒種與大雪窗口處理。

芒種置閏：

- 達超神九日以上後，在芒種後置閏。
- 重走芒種上中下三元，共 15 日。
- 置閏期間 `qimenSolarTerm` 仍為芒種。
- `isIntercalary` 為 `true`。

大雪置閏：

- 達超神九日以上後，在大雪後置閏。
- 重走大雪上中下三元，共 15 日。
- 置閏期間 `qimenSolarTerm` 仍為大雪。
- `isIntercalary` 為 `true`。

置閏結束後，才進入後續奇門節氣局。

## 十三、陰陽遁切換

陰陽遁由 `qimenSolarTerm` 決定。

不由實際夏至 / 冬至交節時間直接決定。

規則：

- 進入奇門夏至上元才開始陰遁。
- 進入奇門冬至上元才開始陽遁。
- 若實際夏至未到，但 `qimenSolarTerm` 已是夏至，仍為陰遁。
- 若實際冬至已到，但 `qimenSolarTerm` 仍是閏大雪，仍為陰遁。

實作時應由 `qimenSolarTerm + yuan` 查局數表取得 `dunType` 與 `ju`，不可由實際交節時間直接切換陰陽遁。

## 十四、跨年處理

resolver 不應只產生單一西曆年 1/1 到 12/31 的 timeline。

原因：

- 大雪置閏會影響後續冬至。
- 冬至又可能影響下一年年初。
- 查詢點若在年初，可能需要前一年冬至後的奇門局序。
- 查詢點若在年底，可能需要下一年冬至前後的緩衝。

建議做法：

- 產生查詢日期前後至少一個冬至循環的 timeline。
- 或產生 query date 前後足夠緩衝，例如前 60 日到後 60 日，再找落點。
- 實作前應先以 2027/12/26 與 2028 年初案例驗證。

跨年比較實際節氣與奇門節氣時，需使用循環序列與時間區間，不要只比較節氣名稱在陣列中的索引。

## 十五、建議實作步驟

建議後續實作順序：

1. 先建立節氣順序常數。
2. 先建立六十甲子順序常數或沿用既有模組。
3. 先建立符頭判定 helper。
4. 先建立元別判定 helper。
5. 先建立指定年份附近的奇門三元局序 timeline。
6. timeline 需跨年，避免年初與年底斷裂。
7. 先用 2027 芒種、大雪案例人工驗證 timeline。
8. 再實作查詢時間落點解析。
9. 最後才接 UI 或 1080 盤查表。

建議不要一開始就把 resolver、UI 與盤面查表綁在一起，避免規則尚未驗證時影響畫面與盤面資料。

## 十六、風險與注意事項

主要風險：

- 把實際節氣與奇門節氣混為同一個欄位。
- 直接用實際夏至 / 冬至切換陰陽遁。
- 把接氣寫成固定天數規則。
- 只產生單年 timeline，導致冬至與跨年錯誤。
- 忘記三元日界線沿用 23:00 換日。
- 從 1080 盤 JSON 反推定局。
- 在規則未驗證前接 UI，造成使用者誤判。

後續測試應先以硬欄位驗證：

- `qimenSolarTerm`
- `yuan`
- `dunType`
- `ju`
- `isIntercalary`

`status` 顯示名稱可先列為 soft 欄位，待狀態命名穩定後再轉為硬驗證。
