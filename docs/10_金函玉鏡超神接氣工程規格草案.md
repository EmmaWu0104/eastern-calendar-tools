## 三、原文可明確推出的工程規則

目前可較安全轉成工程條件的部分：

1. 需要判斷冬至與夏至附近。
2. 換遁與「甲日」有關。
3. 冬至後理論方向為順行，可對應「陽遁」。
4. 夏至後理論方向為逆行，可對應「陰遁」。
5. 不能單純以節氣交接瞬間作為唯一切換點。
6. 存在「超神」狀況：中氣已到，但仍不立即切換。
7. 存在「接氣」狀況：中氣未到，但可提前切換。
8. 自動判斷若無法確定，必須回傳 `manual-required`。

## 四、原文仍有歧義的工程項目

以下項目不可自行實作。

### 1. 「甲日」定義

- 可能是所有天干為甲之日。
- 也可能是甲子旬頭。
- 目前未定。

### 2. 「甲子日在冬至前四日」

- 可能是接氣的具體例子。
- 也可能表示必須甲子日才成立。
- 目前未定。

### 3. 「第四逢之甲」

- 起算點未定。
- 是否含冬至前甲日未定。
- 是第四個甲日或第四日逢甲未定。
- 目前不可自動化。

### 4. 夏至是否完全對稱

- 原文提到冬至、夏至。
- 但示例偏冬至。
- 夏至前後的接氣 / 超神界線需待確認。

### 5. 日界線

- 目前干支曆 23:00 換日。
- 超神接氣是否同樣以 23:00 換日後的日柱判定，需待確認。

### 6. 節氣交接時刻

- 若冬至 / 夏至在一天中間發生，該日整日用哪一遁，需待確認。

## 五、建議函式行為分級

未來 `getJinhanDunType(...)` 可採用三層狀態。

### 第一層：manual-required

用途：

- 規則不足、案例不足、歧義未解時使用。
- 不自動回傳陽遁或陰遁。

回傳範例：

```json
{
  "status": "manual-required",
  "dunType": null,
  "mode": "pending",
  "boundary": null,
  "reason": "超神接氣規則仍有待確認項目，請手動選擇陰陽遁。"
}
```

### 第二層：resolved

用途：

- 規則已能明確判定時使用。
- 回傳陽遁或陰遁。

回傳範例：

```json
{
  "status": "resolved",
  "dunType": "陽遁",
  "mode": "接氣",
  "boundary": "冬至",
  "reason": "依已確認規則判定為冬至接氣，使用陽遁。"
}
```

### 第三層：unsupported

用途：

- 日期超出節氣資料範圍。
- `calendarResult` 不完整。
- `solarTerms` 缺冬至 / 夏至。
- 其他技術上無法判斷的狀況。

回傳範例：

```json
{
  "status": "unsupported",
  "dunType": null,
  "mode": "unknown",
  "boundary": null,
  "reason": "缺少冬至 / 夏至資料，無法判斷金函玉鏡陰陽遁。"
}
```
