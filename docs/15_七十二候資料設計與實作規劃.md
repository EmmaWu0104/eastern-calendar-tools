# 七十二候資料設計與實作規劃

## 一、功能背景

本專案目前已有 24 節氣資料、干支曆核心、四柱查詢、九宮飛星與金函玉鏡日盤。七十二候功能會建立在既有 24 節氣時間之上，先提供每個節氣下的初候、次候、末候資料，後續再做查詢與 UI 顯示。

本包只新增靜態資料與規劃文件，不修改程式邏輯、不修改 UI。

## 二、第一版判斷原則

第一版七十二候判斷原則暫定如下：

- 沿用既有 `data/solar_terms_1899_2101.json` 的 24 節氣交節時間。
- 每個節氣區間以目前節氣交節時間到下一節氣交節時間為範圍。
- 將每個節氣區間三等分為：
  - 初候
  - 次候
  - 末候
- 72 候不受 23:00 換日影響，直接依實際節氣時間區間判斷。
- 第一版不使用真太陽時。
- 第一版不做太陽黃經 5 度精算。

因此第一版七十二候是「節氣區間三等分」版本，不是天文黃經 5 度精算版本。

## 三、資料檔

新增資料檔：

```text
data/seventy_two_hou.json
```

資料以節氣名稱為 key。節氣名稱必須與 `data/solar_terms_1899_2101.json` 使用的節氣名稱一致。

每個節氣固定 3 筆候資料，全年共 24 節氣 x 3 候 = 72 筆。

## 四、欄位說明

每一候包含下列欄位：

- `houIndex`：該節氣內第幾候，範圍為 1 到 3。
- `globalHouIndex`：全年第幾候，範圍為 1 到 72。第一版以立春初候為 1，大寒末候為 72。
- `phase`：候序名稱，固定為初候、次候、末候之一。
- `name`：候名。
- `shortName`：UI 顯示用候名。第一版先與 `name` 相同。
- `description`：說明文字。第一版先保留空字串。

範例：

```json
{
  "立春": [
    {
      "houIndex": 1,
      "globalHouIndex": 1,
      "phase": "初候",
      "name": "東風解凍",
      "shortName": "東風解凍",
      "description": ""
    }
  ]
}
```

## 五、節氣名稱一致性

`data/seventy_two_hou.json` 的 24 個 key 應與既有節氣資料名稱一致：

- 立春
- 雨水
- 驚蟄
- 春分
- 清明
- 穀雨
- 立夏
- 小滿
- 芒種
- 夏至
- 小暑
- 大暑
- 立秋
- 處暑
- 白露
- 秋分
- 寒露
- 霜降
- 立冬
- 小雪
- 大雪
- 冬至
- 小寒
- 大寒

既有 `data/solar_terms_1899_2101.json` 也包含小寒、大寒、立春等相同名稱；資料檢查時應以名稱集合確認，不依 JSON 原始排序推論。

## 六、後續分包規劃

後續建議分包：

- 第 2 包：查表與區間判斷。
  - 新增七十二候查表模組。
  - 載入 `data/seventy_two_hou.json`。
  - 依目前節氣與下一節氣時間三等分判斷目前候。

- 第 3 包：整合節氣查詢結果。
  - 將七十二候結果整合到既有節氣查詢 / `calculateBazi` 相關結果。
  - 確認不影響既有干支曆、金函玉鏡、九宮飛星。

- 第 4 包：UI 顯示目前候與候區間。
  - 在節氣與月令區塊或獨立區塊顯示目前候。
  - 顯示候名、初候 / 次候 / 末候、起訖時間。

- 第 5 包：下一候與 UI 微調。
  - 顯示下一候。
  - 補充候區間進度或簡潔提示。
  - 做基本版面微調。

## 七、本包限制

本包只新增資料與文件：

- 不修改 `src/main.js`。
- 不修改干支曆核心。
- 不修改節氣查詢邏輯。
- 不修改金函玉鏡日盤。
- 不修改九宮飛星。
- 不修改 UI。
- 不新增執行邏輯。
