# 七十二候中日雙版本資料設計與實作規劃

## 一、功能目標

- 七十二候未來支援中國版與日本版候名同時顯示。
- 不改變目前七十二候時間切分公式。
- 仍以目前節氣區間三等分判斷初候 / 次候 / 末候。
- 中國版與日本版只是在同一個候位置上顯示不同物候名稱。
- 第一版不做台灣物候版。
- 第一版不做使用者切換開關，先固定同時顯示中 / 日兩版。

## 二、目前既有規則摘要

- 以目前節氣開始時間到下一節氣開始時間為一個完整節氣區間。
- 將節氣區間三等分：
  - 初候
  - 次候
  - 末候
- 區間採左閉右開。
- `targetDateTime === termStart` 回初候。
- 第一三分點回次候。
- 第二三分點回末候。
- `targetDateTime === nextTermStart` 回新節氣初候，不回上一節氣末候。
- 七十二候不受 23:00 換日影響。
- 不使用真太陽時。

## 三、中國版與日本版的定位

- 中國版：目前既有候名，作為預設 / 傳統曆法語境版本。
- 日本版：同一候位置對應的日本七十二候候名。
- 兩者時間位置相同，候名不同。
- 本工具不判斷哪一版更適合台灣，只負責標記來源並同時顯示。
- 台灣物候版若未來要做，會是第三套候名資料，不屬於本階段。

## 四、建議資料結構

建議將 `data/seventy_two_hou.json` 從單一候名擴充為 `variants` 結構。每一候資料未來可調整為：

```json
{
  "houIndex": 1,
  "globalHouIndex": 1,
  "phase": "初候",
  "name": "東風解凍",
  "shortName": "東風解凍",
  "variants": {
    "zh": {
      "label": "中",
      "name": "東風解凍",
      "shortName": "東風解凍"
    },
    "jp": {
      "label": "日",
      "name": "東風解凍",
      "shortName": "東風解凍"
    }
  }
}
```

- `name` / `shortName` 先保留，並預設指向中國版，避免一次破壞既有 UI 與測試。
- 新增 `variants.zh` 與 `variants.jp`。
- `variants.zh` 放中國版候名。
- `variants.jp` 放日本版候名。
- `label` 用於 UI 顯示來源標記。
- `globalHouIndex` 仍只有一套，不因中日版本拆成兩套。
- 不建議拆成兩份完全獨立 JSON，避免 72 筆候位置對齊困難。
- 若後續真的需要獨立維護來源，也可再評估拆檔，但第一版以單檔 `variants` 為主。

## 五、回傳資料結構規劃

`src/seventyTwoHou.js` 後續應讓 `currentHou` / `nextHou` 帶出 `variants`。建議回傳：

```js
{
  term: "立春",
  phase: "初候",
  houIndex: 1,
  globalHouIndex: 1,

  // 相容舊 UI，預設中國版
  name: "東風解凍",
  shortName: "東風解凍",

  variants: {
    zh: {
      label: "中",
      name: "東風解凍",
      shortName: "東風解凍"
    },
    jp: {
      label: "日",
      name: "東風解凍",
      shortName: "東風解凍"
    }
  },

  start: "...",
  end: "..."
}
```

- 既有 `name` / `shortName` 仍保留。
- 新 UI 可使用 `variants` 顯示雙版本。
- 舊測試可逐步調整，不需要在資料包一次全部改掉。
- 若日本版資料尚未完整填入，`variants.jp` 不可造成程式 throw，可暫時 fallback 到 `name` 或顯示 `—`，實作包時再定。

## 六、UI 顯示規劃

七十二候 UI 未來可改成雙版本顯示。建議格式：

```text
七十二候
中：立春初候・東風解凍
日：立春初候・東風解凍

候區間
02/04 04:02 ～ 02/18 23:51

下一候
中：立春次候・蟄蟲始振
日：立春次候・黄鶯睍睆
```

- `中` 代表中國版。
- `日` 代表日本版。
- 不做切換按鈕，固定雙版本顯示。
- 若日本版候名缺漏，第一版可顯示 `日：—` 或 fallback，實作時需明確決定。
- 手機版要避免擠成一行，建議雙行垂直排列。

## 七、測試規劃

1. 資料完整性測試：
   - 24 節氣皆存在。
   - 每個節氣 3 候。
   - 全年 72 候。
   - `globalHouIndex` 1～72 不重複、不缺漏。
   - 每一候都有 `variants.zh`。
   - 每一候都有 `variants.jp`。
   - 每個 variant 至少有 `name` 或 `shortName`。

2. 相容性測試：
   - 既有 `name` / `shortName` 仍可使用。
   - 預設 `name` / `shortName` 與 `variants.zh` 一致。

3. 邏輯測試：
   - 初候 / 次候 / 末候邊界不變。
   - `nextHou` 判斷不變。
   - 跨年不變。
   - 七十二候仍不受 23:00 換日影響。

4. UI 驗收：
   - `currentHou` 顯示中 / 日兩版。
   - `nextHou` 顯示中 / 日兩版。
   - 候區間顯示不受影響。
   - 手機版不擠壞。
