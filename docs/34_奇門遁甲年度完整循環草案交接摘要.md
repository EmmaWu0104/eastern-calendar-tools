# 34 奇門遁甲年度完整循環草案交接摘要

## 1. 本階段定位

本階段是傳統置閏法 resolver 的草案 pipeline 收斂。目前仍未接 UI，未接 1080 盤面 JSON，未擴大 `INITIAL_QIMEN_TIMELINE`，未改 `resolveQimenJu` 主流程，也未取代既有 2027 fixture / 初版 resolver。

這一階段只建立「年度窗口分析 -> full cycle draft input -> full cycle timeline draft」的 helper 與測試，作為後續完整年度奇門 timeline 的 future path。

## 2. 第 25 包完成內容

```js
buildQimenFullTermCycleDraftInputForYear(year, options = {})
```

此 helper 使用 `analyzeQimenIntercalationWindowsForYearAuto(year)`，第一版只支援 `startTerm = "大雪"`，回傳：

- `year`
- `startSeed`
- `intercalations`
- `windows`

2027 案例：

- `startSeed` = 大雪 `2027-11-25T23:00:00+08:00`
- `intercalation` = 大雪 `2027-12-10T23:00:00+08:00`
- `windows` = 芒種 / 大雪

此 helper 不自行重新判斷置閏日數，只使用年度窗口分析結果，也不接 resolver 主線。

## 4. 目前 helper 關係

草案 pipeline：

```text
analyzeQimenIntercalationWindowsForYearAuto(year)
-> buildQimenFullTermCycleDraftInputForYear(year)
-> buildQimenTimelineFromFullTermSeedCycle(...)
-> buildQimenFullTermCycleTimelineDraftForYear(year)
```

既有主線仍是：

```text
buildQimenYearSeedRecommendations(2027)
-> buildQimenTimelineFromYearSeedRecommendations(2027)
-> buildSeedDrivenQimenTimelineFixture2027()
-> INITIAL_QIMEN_TIMELINE
-> resolveQimenJu(...)
```

草案 pipeline 尚未取代主線。目前它只是可測試、可比對的 future path。

草案串接 pipeline：

```text
buildQimenFullTermCycleTimelineDraftForYear(year)
-> buildQimenMultiYearFullTermCycleTimelineDraft({ startYear, endYear })
-> diagnostics duplicate / gap / overlap check
```

多年串接 helper 尚未取代主線。它只是 future resolver timeline window 的候選基礎。

lookup 草案 pipeline：

```text
findQimenFullTermCycleTimelineDraftEntry(dateTimeText)
-> getQimenEffectiveDayStart(dateTimeText)
-> candidateYear / candidateYear - 1 / candidateYear + 1
-> buildQimenFullTermCycleTimelineDraftForYear(year)
-> selected timeline entry
```

這條 pipeline 尚未接 `resolveQimenJu()`。它只是 future resolver replacement 的 lookup strategy 實驗。

### 8.6 read-only lookup strategy 草案

第 38 包新增 read-only helper：

```js
findQimenFullTermCycleTimelineDraftEntry(dateTimeText, options = {})
```

第一版支援：

```js
{
  strategy: "cycle-year"
}
```

用途：

- 依查詢時間找出 full cycle draft timeline entry。
- 不使用 `INITIAL_QIMEN_TIMELINE`。
- 不影響 `resolveQimenJu()`。
- 不接 UI。
- 不讀取 1080 盤面 JSON。
- 作為後續 resolver 替換前的 lookup strategy 草案。

#### 8.6.1 cycle-year strategy

策略流程：

1. 使用 `getQimenEffectiveDayStart(dateTimeText)` 取得奇門有效日 23:00 起點。
2. 以查詢時間的台北 civil year 作為 candidate year。
3. 先查 candidate year 的 `buildQimenFullTermCycleTimelineDraftForYear(year)`。
4. 若該 yearDraft 覆蓋查詢有效日，回傳該 entry。
5. 若沒有命中，再依序嘗試 `candidateYear - 1`、`candidateYear + 1`。
6. 若仍沒有命中，回傳 `null`。
7. 若 strategy 不支援，throw `RangeError`。
8. 若年份資料缺失，沿用底層 helper 的 `RangeError`。

回傳 metadata：

```js
lookup: {
  strategy: "cycle-year",
  queryEffectiveDayStart,
  selectedYear,
  candidateYears
}
```

`candidateYears` 是實際嘗試過的年份。這個 helper clone entry 後回傳，不修改原始 timeline entry。

- 已知第一筆 duplicate 差異為 `qimenSolarTerm`：1909 年 cycle 視為小雪，1910 年 cycle 視為大雪。
