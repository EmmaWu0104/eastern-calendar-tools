# 奇門遁甲資料檔格式

本文件記錄奇門遁甲第二包新增資料檔的格式。本包只建立資料格式與骨架，不代表奇門定局 resolver 或完整盤面已實作。

## 一、本文件目的

本文件目的，是讓後續奇門遁甲功能在補資料、寫查表模組與接 UI 前，先固定資料檔邊界。

本包只處理：

- 二十四節氣三元局數表。
- 1080 盤 JSON 拆檔骨架。
- `null` 盤面佔位。
- 資料完整性測試。

本包不實作置閏法 resolver。

## 三、二十四節氣三元局數表格式

`data/qimen/qimen_yuan_ju_table.json` 只保存局數表，不判斷超神、接氣、置閏。

格式：

```json
{
  "meta": {
    "schemaVersion": "1.0.0",
    "name": "奇門遁甲二十四節氣三元局數表",
    "rule": "置閏法",
    "notes": "陰陽遁切換由奇門節氣局序決定，不直接由實際夏至或冬至交節時刻決定。"
  },
  "terms": {
    "冬至": {
      "dunType": "yang",
      "ju": {
        "上元": 1,
        "中元": 7,
        "下元": 4
      }
    }
  }
}
```

資料規則：

- 節氣名稱使用繁體中文。
- `dunType` 只允許 `yang` 或 `yin`。
- 元別 key 只允許 `上元`、`中元`、`下元`。
- 局數只允許 1 到 9 的整數。
- 不加入 `method` 欄位。

## 四、1080 盤 JSON 拆檔策略

1080 盤採 JSON 查表，不由程式直接推盤。

拆檔方式：

```text
陽遁 9 局 x 60 時辰盤 = 540 盤
陰遁 9 局 x 60 時辰盤 = 540 盤
合計 1080 盤
```

因此拆為 18 檔：

- `data/qimen/plates/yang/ju-1.json` 到 `data/qimen/plates/yang/ju-9.json`
- `data/qimen/plates/yin/ju-1.json` 到 `data/qimen/plates/yin/ju-9.json`

18 個 plates JSON 只保存盤面資料，不判斷定局。

定局欄位由後續 resolver 回傳，不寫入 plates JSON。

## 五、單一盤面 JSON 骨架格式

以 `data/qimen/plates/yang/ju-1.json` 為例：

```json
{
  "meta": {
    "schemaVersion": "1.0.0",
    "dunType": "yang",
    "dunName": "陽遁",
    "ju": 1,
    "plateCount": 60,
    "source": "御定奇門遁甲寶鑑整理",
    "notes": "本檔為盤面資料骨架。null 代表該時辰盤面尚未建立。"
  },
  "plates": {
    "甲子": null,
    "乙丑": null,
    "丙寅": null
  }
}
```

實際檔案中 `plates` 必須包含完整 60 甲子時辰 key。

## 六、null 的意義

plates 中時辰 key 為 `null`，代表該盤尚未建立。

本包所有 plates value 皆為 `null`。

UI 或查表模組遇到 `null` 時，應顯示：

```text
盤面資料尚未建立，目前僅顯示定局結果
```

遇到 `null` 不應視為資料檔錯誤，也不應顯示空白九宮格。

## 七、後續盤面補齊格式預告

後續正式盤面資料應逐步替換 `null`。

正式盤面物件可再另開文件固定 schema，例如：

```json
{
  "meta": {
    "hourPillar": "甲子"
  },
  "palaces": {
    "坎": {},
    "艮": {},
    "震": {},
    "巽": {},
    "離": {},
    "坤": {},
    "兌": {},
    "乾": {},
    "中": {}
  }
}
```

盤面 JSON 不應保存：

- `actualSolarTerm`
- `qimenSolarTerm`
- `status`
- `yuan`
- 陰陽遁 / 局數 resolver 判斷結果
- 超神、接氣、置閏判斷結果

這些定局欄位應由奇門定局 resolver 負責。

## 八、測試規則

`tests/run-tests.js` 應檢查：

- `qimen_yuan_ju_table.json` 存在且可解析。
- 三元局數表剛好包含 24 個節氣。
- 24 個節氣名稱完整。
- 冬至到芒種為 `yang`。
- 夏至到大雪為 `yin`.
- 每個節氣都有 `上元`、`中元`、`下元` 局數。
- 局數符合 `docs/27_奇門超神接氣置閏規則.md` 的表。
- 18 個 plates JSON 存在且可解析。
- plates 檔案 `meta.dunType` 與路徑一致。
- `meta.dunName` 與 `dunType` 一致。
- `meta.ju` 與檔名一致。
- `meta.plateCount` 為 60。
- `plates` 剛好有 60 個 key。
- 60 個 key 與 60 甲子完整一致。
- 本包新建骨架全部為 `null`。

測試只檢查資料檔結構，不檢查定局演算法。

## 九、本包限制

本包不處理：

- 置閏法 resolver。
- 超神、接氣、置閏演算法。
- UI。
- 奇門區塊顯示。
- 正式盤面資料。
- 從 JSON 推算定局規則。
- 既有日盤。
- 金函玉鏡。
- 九宮飛星。
- 每日資訊。
- 既有 24 節氣 JSON。
- 既有干支曆核心。

本包只建立資料骨架與資料完整性測試。
