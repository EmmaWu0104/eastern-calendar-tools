# 47 奇門遁甲 1080.md 解析與檢查規劃

本文件是新第 84 包 docs-only 規劃文件，目標是規劃如何從 `data/1080.md` 解析 1080 盤。

本包不實作 parser，不修改 `data/qimen/plates/**`，不填入任何 plate JSON，不做 UI rendering，也不做格局、卦名、斷語、吉凶。本包只規劃解析規則、標準化規則與診斷檢查規則。

若發現 `data/1080.md` 內容疑似錯誤，後續應先列報告給使用者修正，再重新匯入。

## 1. 資料來源與定位

`data/1080.md` 是人工整理的奇門 1080 盤原始資料。

內容包含：

* 陽遁一局～九局
* 陰遁九局～一局，或依文件實際順序
* 每局 60 個時柱盤

每盤以 markdown table 呈現。

目前只取基本盤面資料：

* 時柱
* 值符星
* 值使門
* 九宮
* 天盤干
* 地盤干
* 九星
* 八門
* 八神

不處理：

* 格局
* 卦名
* 斷語
* 吉凶
* 空亡
* 馬星
* 三奇六儀格局
* 其它流派補充

補充：

* `data/1080.md` 是轉換來源，不是 runtime lookup 直接使用資料。
* runtime 最終仍應讀 `data/qimen/plates/{dunType}/ju-{ju}.json`。
* 正式轉入 JSON 前要先通過解析與診斷檢查。

## 2. markdown 結構規則

parser 未來可依下列結構解析。

### 2.1 遁別標題

```md
# 陽遁
# 陰遁
```

對應：

* `陽遁` -> `dunType: "yang"`、`dunName: "陽遁"`
* `陰遁` -> `dunType: "yin"`、`dunName: "陰遁"`

### 2.2 局數標題

```md
### 陽遁一局
### 陰遁九局
```

對應：

* 中文數字一～九轉為 `ju: 1`～`ju: 9`
* 標題中的遁別必須與目前 `# 陽遁` / `# 陰遁` context 一致

### 2.3 日群標題

```md
##### 甲己日
##### 乙庚日
##### 丙辛日
##### 丁壬日
##### 戊癸日
```

用途：

* 作為資料定位與檢查用
* 不一定寫入 plate object
* 可用於診斷某時柱是否落在預期日群下

### 2.4 盤面表格

每盤由 markdown table 表示：

```md
| 甲子 | | 直符：蓬<br>直使：休 |
| --- | --- | --- |
| ... | ... | ... |
| ... | ... | ... |
| ... | ... | ... |
```

解析規則：

* 第一列第一格為 `hourPillar`
* 第一列第三格包含：
  * `直符：{starShort}`
  * `直使：{doorShort}`
* 接下來三列為九宮資料
* 第一個資料列對應：
  * 巽、離、坤
* 第二個資料列對應：
  * 震、中、兌
* 第三個資料列對應：
  * 艮、坎、乾

## 3. 九宮 layout 對應

markdown table 到 palace key 的 mapping：

| table row | col 1 | col 2 | col 3 |
|---|---|---|---|
| 第一宮列 | `xun` 巽 | `li` 離 | `kun` 坤 |
| 第二宮列 | `zhen` 震 | `center` 中 | `dui` 兌 |
| 第三宮列 | `gen` 艮 | `kan` 坎 | `qian` 乾 |

每宮固定 meta：

* `xun`：巽、東南、4
* `li`：離、南、9
* `kun`：坤、西南、2
* `zhen`：震、東、3
* `center`：中、中、5
* `dui`：兌、西、7
* `gen`：艮、東北、8
* `kan`：坎、北、1
* `qian`：乾、西北、6

補充：

* markdown 的 row/col 順序是 UI layout，不是 object key 順序。
* 輸出 JSON 可以使用既有 `QIMEN_PALACE_KEYS` 順序。
* UI rendering 未來再決定是否用 layout array 顯示。

## 4. 宮格文字解析規則

### 4.1 一般宮格

一般宮格格式：

```text
辛 輔<br>辛 杜 合
```

解析為：

* 第一行：
  * `heavenStem: "辛"`
  * `star: "天輔"`
* 第二行：
  * `earthStem: "辛"`
  * `door: "杜"`
  * `deity: "六合"`

補充：

* `<br>` 左右可能有空白。
* 欄位之間可能有多個空白。
* parser 應先 normalize whitespace。
* 若 token 數不足或過多，應列 diagnostic，不要靜默修正。

### 4.2 中宮

中宮常見格式：

```text
壬 禽<br>壬
```

解析為：

* `heavenStem: "壬"`
* `star: "天禽"`
* `earthStem: "壬"`
* `door: null`
* `deity: null`

補充：

* 中宮一般不應有八門。
* 中宮一般不應有八神。
* 若中宮出現門或神，先列 warning 或 error，需人工確認。
* 中宮仍需保留 palace meta。

### 4.3 特殊格式與疑似錯字

* 有些表格可能出現格式不齊、空格不一致、字貼在一起、或 OCR / 人工整理錯字。
* parser 第一版不應自動猜太多。
* 可以做有限標準化，但要保留 diagnostics。
* 疑似錯誤應列出檔案位置、遁別、局數、時柱、宮位、原文字串與錯誤原因。

## 5. 簡稱標準化規則

未來 converter 使用下列標準化 mapping。

### 5.1 九星

| 簡稱 | 標準值 |
|---|---|
| 蓬 | 天蓬 |
| 任 | 天任 |
| 衝 | 天衝 |
| 輔 | 天輔 |
| 英 | 天英 |
| 芮 | 天芮 |
| 柱 | 天柱 |
| 心 | 天心 |
| 禽 | 天禽 |

補充：

* `直符：蓬` 應標準化為 `zhiFuStar: "天蓬"`。
* 宮格內 `輔` 應標準化為 `star: "天輔"`。
* 若出現 `逢`，高度疑似 `蓬` 的誤字，應列 diagnostic，不建議自動修正為正式資料。
* 若使用 auto-fix，必須在報告中標記。

### 5.3 八神

補充：

* 若出現非八神字，應列 diagnostic。

## 6. plate object 輸出目標

未來每張盤輸出為第 79～81 包定義的 plate object：

```json
{
  "schemaVersion": 1,
  "hourPillar": "甲子",
  "zhiFuStar": "天蓬",
  "zhiShiDoor": "休",
  "xunShou": null,
  "notes": [],
  "palaces": {
    "kan": {},
    "kun": {},
    "zhen": {},
    "xun": {},
    "center": {},
    "qian": {},
    "dui": {},
    "gen": {},
    "li": {}
  }
}
```

每宮輸出：

```json
{
  "palaceName": "坎",
  "direction": "北",
  "luoshuNumber": 1,
  "earthStem": "戊",
  "heavenStem": "戊",
  "door": "休",
  "star": "天蓬",
  "deity": "值符",
  "isEmpty": false,
  "isHorse": false,
  "isZhiFuPalace": true,
  "isZhiShiPalace": true,
  "notes": []
}
```

補充：

* `isZhiFuPalace` 可由 `star === zhiFuStar` 判斷。
* `isZhiShiPalace` 可由 `door === zhiShiDoor` 判斷。
* 若多宮符合或沒有宮符合，應列 diagnostic。
* `xunShou` 第一版可維持 null，除非 `1080.md` 有明確資料來源。

## 7. 資料完整性檢查規則

parser / diagnostic 應檢查下列項目。

### 7.1 檔案層

* 必須同時有 `# 陽遁` 與 `# 陰遁`。
* 每個遁別應有 9 局。
* 每個局應有 60 盤。
* 理想總數為 1080 盤。
* 每個 `dunType + ju + hourPillar` 不可重複。
* 每個 `dunType + ju` 應剛好有 60 個時柱。

### 7.2 時柱層

* 每局需包含 60 甲子時柱。
* 時柱文字必須符合 60 甲子。
* 應正規化常見異體：
  * `戍` 高度疑似 `戌`
  * `壬 戌` 多空白應正規化為 `壬戌`
* 但正規化應列入 diagnostics。
* 若時柱缺漏或重複，應列 error。

### 7.3 每盤層

每盤應：

* 有 `hourPillar`
* 有 `zhiFuStar`
* 有 `zhiShiDoor`
* 有 9 宮資料
* 每宮可解析
* 非中宮 8 宮應有：
  * 天盤干
  * 地盤干
  * 九星
  * 八門
  * 八神
* 中宮應有：
  * 天盤干
  * 地盤干
  * 天禽
  * 無八門
  * 無八神

### 7.4 唯一性檢查

每盤理想上：

* 九星：
  * 八個外宮星 + 中宮天禽
  * 或視資料格式統計 9 宮 star
* 八門：
  * 非中宮八門各一次
* 八神：
  * 非中宮八神各一次
* 天盤干：
  * 乙、丙、丁、戊、己、庚、辛、壬、癸各一次
* 地盤干：
  * 乙、丙、丁、戊、己、庚、辛、壬、癸各一次

補充：

* 使用者目前期待每張盤每顆元素只有一個。
* 若出現重複或缺漏，應列 error。
* 中宮也納入天盤干 / 地盤干統計。
* 八門 / 八神不含中宮。

## 8. 錯誤與警告分類

diagnostics 等級規劃如下。

### error

* 遁別缺失
* 局數缺失
* 某局不是 60 盤
* 時柱缺失
* 時柱重複
* 時柱不在 60 甲子
* 表格不是 3x3
* 宮格無法解析
* 非中宮缺天盤干 / 地盤干 / 九星 / 八門 / 八神
* 中宮缺天盤干 / 地盤干 / 天禽
* 每盤八門重複或缺漏
* 每盤八神重複或缺漏
* 每盤天盤干重複或缺漏
* 每盤地盤干重複或缺漏
* `zhiFuStar` 找不到對應宮
* `zhiShiDoor` 找不到對應宮

### warning

* 常見異體字正規化
* `戍` -> `戌`
* 多餘空白
* `<br>` 前後格式不一致
* 疑似 `逢` -> `蓬`
* 宮格 token 順序可解析但不標準
* 中宮出現額外 token
* 來源 table 周圍有空白表格或異常空列

補充：

* error 應阻止寫入正式 JSON。
* warning 可產出報告，但是否阻止寫入需人工決定。
* 第一版建議只要有 warning，也先不要自動寫正式 JSON，等使用者確認。

## 9. 診斷報告格式

未來 parser diagnostic result：

```js
{
  ok: false,
  stats: {
    totalPlates: 1080,
    yangPlates: 540,
    yinPlates: 540,
    errors: 0,
    warnings: 0
  },
  errors: [],
  warnings: []

```

每筆 diagnostic：

```js

  level: "error",
```

* `MISSING_DUN_SECTION`
* `MISSING_JU_SECTION`
* `PLATE_COUNT_MISMATCH`

* `INVALID_HOUR_PILLAR`
* `NORMALIZED_HOUR_PILLAR`
* `INVALID_TABLE_SHAPE`
* `INVALID_CELL_FORMAT`
* `UNKNOWN_STAR`
* `UNKNOWN_DOOR`
* `UNKNOWN_DEITY`
* `UNKNOWN_STEM`
* `DUPLICATE_STAR`
* `MISSING_STAR`
* `DUPLICATE_DOOR`
* `MISSING_DOOR`
* `DUPLICATE_DEITY`
* `MISSING_DEITY`
* `DUPLICATE_HEAVEN_STEM`
* `MISSING_HEAVEN_STEM`

## 10. 輸出策略

### 10.1 先只產生 diagnostic，不輸出 JSON

優點：

* 最安全。
* 可以先讓使用者修 `1080.md`。
* 不會污染 `data/qimen/plates/**`。
* 適合第一包 parser。

建議：

* 第 85 包先採此方案。

### 10.2 parser 直接產生 plate JSON

優點：

* 快速完成資料轉換。
* 可直接進 validation。

缺點：

* 若 `1080.md` 有錯，會快速污染 18 個正式 JSON。
* diff 會非常大。
* 不利人工審查。

建議：

* 等 diagnostic 乾淨後再做。
* 不應與 parser 初版同包。

## 13. 本文件結論

* `data/1080.md` 可以作為 1080 盤轉換來源。
* 但在寫入正式 JSON 前，必須先做 parser diagnostics。
* 第一階段重點是找出盤面資料錯誤、格式錯誤、重複與缺漏。
* 有錯先列清單給使用者修正。
* 盤面資料乾淨後再轉入 `data/qimen/plates/**`。
* 格局、卦名與斷語後續另包討論。
