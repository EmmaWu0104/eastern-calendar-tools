## 3. 完整資料流

目前完整流程：

```text
data/1080.md
-> parseQimen1080Markdown
-> convertQimen1080ParsedToDryRun
-> buildQimen1080FormalPlateAdapterReport
-> formal candidate writer
-> data/qimen/plates/{yang,yin}/ju-{1..9}.json
-> getQimenPlate({ dunType, ju, hourPillar })
-> renderQimenSection
-> 奇門盤面 UI 九宮格
```

## 4. UI 渲染資料來源

UI 渲染資料來源與邊界：

* resolver 仍使用 `resolveQimenJuFromFullTermCycleDraft`。
* UI 由 resolver 取得 `dunType` / `ju` / `hourPillar`。
* UI 呼叫 `getQimenPlate`。
* UI 不直接讀 JSON。
* UI 不讀 preview / candidate output。
* 手動覆寫只改盤面查表 `dunType` / `ju`，不改左側自動定局摘要。
* `getQimenPlate` implementation 未修改。

UI 顯示內容摘要：

* 九宮順序：巽、離、坤 / 震、中、兌 / 艮、坎、乾。
* 每宮顯示：
  * 宮名 / 方位 / 洛書數。
  * 天盤干。
  * 地盤干。
  * 九星。
  * 八門。
  * 八神。
* badge：
  * 直符。
  * 直使。
* 中宮：
  * 顯示天禽。
  * null 欄位顯示 fallback，不顯示 `null` 字樣。
* `source.raw` 不顯示在正式 UI。
